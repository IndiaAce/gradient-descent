Prompt chaining.

One model call's output becomes the next call's input. Instead of one bloated prompt doing five things badly, you build focused links — each with one job, each independently loggable, retryable, and swappable. The reliability lives in the seams between calls. Introduced this edition as cycle 1 of the agent-engineering ladder.

§ T-01 · WHY IT MATTERS
CORE THESIS · DETECTION CONTEXT

Three small calls beat one big one.

Ask a model to extract, analyze, summarize, critique, and format in a single prompt and you get a mush where each step half-fails and you can't tell which. Chaining breaks the work into links you can actually reason about.

Because each link is a separate API call, the seams are where the control lives: log every step, evaluate it against criteria, retry on failure, or swap a cheaper model for the easy links and a stronger one for the critique. The canonical shape is self-correction — generate, critique, rewrite.

For a detection engineer, the win is auditability. When an analyst disputes a generated threat note, you can point to the exact link that produced the weak sentence — not shrug at a black box.

When to reach for it:

  • Self-correction — draft → critique → rewrite
  • Pipelines — extract → transform → summarize
  • Validation gates — generate, then verify before use
  • Model mixing — cheap links + one strong critique

Reach-for-it patterns.

// PATTERN 01 · LINEAR CHAIN

One helper, three calls, each output feeding the next. The simplest reliable shape. Source: §C-04 ED #006.

python · pattern 01 · linear chain // 3 LINKS 
def ask(prompt: str) -> str:        # one link = one call
    r = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=400,
        messages=[{"role": "user", "content": prompt}],
    )
    return r.content[0].text

plain  = ask(f"Translate to English:\n{drivers}")
draft  = ask(f"3-sentence note from:\n{plain}")
report = ask(f"Cut hedging, rewrite tighter:\n{draft}")
