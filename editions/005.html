MLflow Tracking GBM

MLflow: stop guessing, start comparing.

You have a working GBM hitting 98% accuracy. But which n_estimators got you there? What max_depth did you finally settle on? When you retrain on fresh labeled data next month, how will you know the new run is actually better — or just differently broken?

Without experiment tracking, model development is archaeology — reconstructing what past-you did from notebook cells and vague memory. MLflow fixes this. Every training run becomes a named, permanent record with its exact parameters, evaluation metrics, and saved model artifact.

The API surface you need is tiny: set_experiment() names the project, start_run() opens a record, three logging calls fill it in. Comparison views, registry, artifact browser — free.

Change one param, re-run with a different run_name, and the UI lets you diff them. That's the entire loop.

// see also Full topic page on MLflow with every appearance across editions, the full code archive, and the roadmap of where it goes next — topics/mlflow →
python · baseline_gbm.py // FIG C-02-α 
import mlflow
import mlflow.sklearn
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import roc_auc_score, f1_score

params = {"n_estimators": 100, "max_depth": 3, "learning_rate": 0.1}

mlflow.set_experiment("calendra-gbm")

with mlflow.start_run(run_name="baseline"):
    model = GradientBoostingClassifier(**params, random_state=42)
    model.fit(X_train, y_train)

    mlflow.log_params(params)
    mlflow.log_metric("auc_roc", roc_auc_score(y_test, model.predict_proba(X_test)[:, 1]))
    mlflow.log_metric("f1",      f1_score(y_test, model.predict(X_test)))
    mlflow.sklearn.log_model(model, "model")

# Now run:  mlflow ui  →  open localhost:5000

Tool Use Anthropic SDK

Tool use, the function-dispatch loop.

CYCLE 0 / 7 L3 · LLM ENG EDITION #005 SDK: anthropic-py 0.42+

Tool use is the pattern where you give Claude a set of functions it can call during a response — pausing generation to invoke them and incorporating their return values before continuing.

You describe each tool with a name, description, and JSON schema for its inputs. Claude decides when to call it. You implement the logic. The loop runs until Claude reaches a natural stopping point. This isn't prompt engineering — it's a proper function-dispatch loop with Claude as the decision-maker about when to invoke external systems.

The most immediate use case for you: a Claude agent that reads a Calendra verdict JSON and writes a human-readable threat report, optionally calling lookup tools — domain age, URL category — for context the verdict alone can't carry.

NEXT UP · CYCLE 1 / 7 Prompt chaining — when a single call isn't enough and Claude's output needs to become the input for the next step.
python · threat_report_agent.py // FIG C-04-α 
import anthropic, json, os

client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

tools = [{
    "name": "lookup_domain_age",
    "description": "Returns approximate age of a domain in years",
    "input_schema": {
        "type": "object",
        "properties": {"domain": {"type": "string"}},
        "required": ["domain"],
    },
}]

def write_threat_report(verdict: dict) -> str:
    messages = [{"role": "user", "content":
        f"Write a 3-sentence threat report. Verdict: {json.dumps(verdict)}"}]

    while True:
        resp = client.messages.create(
            model="claude-sonnet-4-6", max_tokens=512,
            tools=tools, messages=messages,
        )
        if resp.stop_reason == "end_turn":
            return next(b.text for b in resp.content if hasattr(b, "text"))
        results = [execute_tool(b.name, b.input)
                   for b in resp.content if b.type == "tool_use"]
        messages += [{"role": "assistant", "content": resp.content},
                     {"role": "user",      "content": results}]

IDEA #1 · L2 · ~2 HRS · CALENDRA
Embeddings Features

Trade url_count for a description embedding.

// IDEA #1 OF 2 THIS EDITION HYPOTHESIS: description embeddings > raw URL count on AUC REV 005-A

Your Calendra feature pipeline uses url_count as a raw integer. Useful, but brittle: a single obfuscated link scores the same as a zero-URL benign invite, and a verbose phishing description with no URLs at all gets ignored entirely.

The description field itself contains far richer signal. Attackers write things like "Click here to confirm your attendance: http://malcal-invite.xyz/rsvp?t=abc123" — the surrounding language is semantically suspicious before you ever parse the URL.

Sentence transformers map raw text into a 384-dim vector encoding semantic meaning. Add cosine similarity to a centroid of known-malicious description embeddings as a single new feature — or use the raw embedding dims via PCA, or test both alongside your existing features with MLflow.

↳ CONNECTS TO §C-03 BUILD — log the embedding-augmented run alongside your baseline in MLflow and compare AUC + F1 directly in the UI. topics/embeddings →
python · description_embed.py // FIG C-05-α 
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity

model = SentenceTransformer("all-MiniLM-L6-v2")  # 384-dim, fast

descriptions = [
    "Join the team for our Q2 planning session — Google Meet link inside",
    "Confirm your attendance: http://quarterly-sync.xyz/rsvp?token=abc",
]
embeddings = model.encode(descriptions)   # shape: (n, 384)

# Cosine similarity to a centroid of known-malicious embeddings
mal_centroid   = embeddings[labels == 1].mean(axis=0, keepdims=True)
suspicion_feat = cosine_similarity(embeddings, mal_centroid).flatten()
