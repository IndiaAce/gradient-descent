§ C-02 · CONCEPT → CODE
~ 7 MIN · L1 PYTHON · REF: sklearn 1.8
Pipelines ColumnTransformer GBM

Pipelines: one object, zero leakage.

Right now Calendra's preprocessing probably lives as loose cells: scale these columns, one-hot those, then call .fit(). The bug hiding in that flow is data leakage — if you scale on the full dataset before splitting, your test fold has already peeked at the training set's statistics. Your reported AUC is optimistic fiction.

A Pipeline fixes this structurally. It chains preprocessing and model into one estimator. When cross-validation re-fits it on each fold, the scaler learns its means from that fold's training rows only. Leakage becomes impossible by construction, not by discipline.

ColumnTransformer is the other half: different columns, different treatment. Numeric features get scaled; categorical TLDs and languages get one-hot encoded with handle_unknown="ignore" so an unseen .zip domain at inference produces all-zeros instead of a crash.

The payoff: the whole thing pickles as one .joblib. Preprocessing travels with the model — no separate scaler to version, lose, or mismatch at inference.

// see also Full topic page on Pipelines — every appearance, the growing code archive, and the roadmap from baseline wrap to full feature-union — topics/pipelines →
python · leakproof_pipeline.py // FIG C-02-α 
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import train_test_split

numeric = ["url_count", "sender_domain_age", "content_len", "attendee_count"]
nominal = ["organizer_tld", "detected_language"]

pre = ColumnTransformer([
    ("num", StandardScaler(), numeric),                       # scale on fold-train stats only
    ("cat", OneHotEncoder(handle_unknown="ignore"), nominal), # unseen TLD -> all-zeros
])

pipe = Pipeline([
    ("pre", pre),                                          # preprocessing re-fit per fold
    ("gbm", GradientBoostingClassifier(random_state=42)),
])

X_tr, X_te, y_tr, y_te = train_test_split(X, y, stratify=y, random_state=42)
pipe.fit(X_tr, y_tr)            # scaler learns means from X_tr ONLY
print(pipe.score(X_te, y_te))  # one estimator: preprocessing + model, no leak

python · build_pipeline.py // FIG C-03-α 
import joblib
import pandas as pd
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import cross_val_score, train_test_split

# 1 — load the labeled ICS feature table (one row per invite)
df = pd.read_parquet("data/train.parquet")   # TODO: your real path
y  = df.pop("is_malicious")                   # label column -> target

# 2 — declare which columns get which treatment
NUMERIC = ["url_count", "sender_domain_age", "content_len", "attendee_count"]
NOMINAL = ["organizer_tld", "detected_language"]
# TODO: distribute the rest of your 12 features into the right list

# 3 — one ColumnTransformer = all preprocessing, re-fit per fold
pre = ColumnTransformer([
    ("num", StandardScaler(), NUMERIC),
    ("cat", OneHotEncoder(handle_unknown="ignore"), NOMINAL),
    # TODO: ("txt", TfidfVectorizer(max_features=300), "subject")  -> stretch
])

# 4 — glue preprocessing + estimator into a single object
pipe = Pipeline([
    ("pre", pre),
    ("gbm", GradientBoostingClassifier(n_estimators=200, random_state=42)),
])

# 5 — honest score: CV runs preprocessing INSIDE each fold
X = df[NUMERIC + NOMINAL]
cv = cross_val_score(pipe, X, y, cv=5, scoring="roc_auc")
print(f"5-fold AUC: {cv.mean():.4f} +/- {cv.std():.4f}")

# 6 — fit on a split, persist the WHOLE pipeline as one artifact
X_tr, X_te, y_tr, y_te = train_test_split(X, y, stratify=y, random_state=42)
pipe.fit(X_tr, y_tr)
joblib.dump(pipe, "models/calendra_pipeline.joblib")  # scaler travels with model

# 7 — reload + predict: no manual scaling at inference, ever again
reloaded = joblib.load("models/calendra_pipeline.joblib")
print(reloaded.predict(X_te.head()))   # TODO: wire into FastAPI /predict

§ C-04 · AGENT BRIEF

Prompt Chaining Anthropic SDK

Prompt chaining, one link at a time.

CYCLE 1 / 7 L3 · LLM ENG EDITION #006 SDK: anthropic-py 0.42+

Prompt chaining is the pattern where one model call's output becomes the next call's input. Instead of asking Claude to do five things in a single prompt — and getting a mush where each step half-fails — you break the work into focused links, each with one job.

The reliability comes from the seams. Every link is a separate API call, so you can log it, evaluate it, retry it, or swap its model independently. The most common shape is self-correction: generate a draft, critique the draft against criteria, rewrite from the critique.

For you: take a Calendra verdict's SHAP attributions, translate them to plain English, draft a three-sentence analyst note, then have a critique link strip the hedging and tighten it. Each step is inspectable — when an analyst disputes a report, you can see exactly which link produced the weak sentence.

NEXT UP · CYCLE 2 / 7 Multi-step orchestration — when the chain stops being a straight line and a controller decides which link runs next.
python · verdict_chain.py // FIG C-04-α 
import anthropic, os

client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

def ask(prompt: str) -> str:          # one link in the chain = one call
    r = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=400,
        messages=[{"role": "user", "content": prompt}],
    )
    return r.content[0].text

# SHAP drivers -> plain English -> analyst draft -> self-critique rewrite
drivers = "url_count=+0.41, sender_domain_age=+0.33, organizer_tld=.xyz=+0.22"

plain  = ask(f"Translate these SHAP attributions into plain English:\n{drivers}")
draft  = ask(f"Write a 3-sentence analyst threat note from:\n{plain}")
report = ask(f"Cut hedging and vague verbs, then rewrite tighter:\n{draft}")

print(report)   # every link is loggable + independently swappable

Optuna GBM

Let Optuna tune the whole pipeline.

// IDEA #2 OF 2 THIS EDITION HYPOTHESIS: a 50-trial search beats your hand-picked params REV 006-A

Your GBM's n_estimators=100, max_depth=3, learning_rate=0.10 were hand-picked across a handful of runs. That's a guess dressed as a decision. Optuna turns it into a search: define the ranges, let a Bayesian sampler propose 50 configs, keep the best by cross-validated AUC.

The trick is to tune the Pipeline you just built in §C-03, not a bare model. Optuna addresses nested params with the step__param syntax — gbm__max_depth reaches inside the pipeline. Because the objective wraps cross_val_score, every trial is leak-free by construction.

Wire each trial to an MLflow child run (per §C-01) and the search tree becomes the same sortable history you set up last edition. Lock the winner as v5.0.

↳ CONNECTS TO §C-03 BUILD — Yes. The Optuna objective tunes the exact cross-validated Pipeline you packaged this weekend. topics/mlflow →
python · tune_pipeline.py // FIG C-05-α 
# pip install optuna
import optuna
from sklearn.model_selection import cross_val_score

def objective(trial):                          # one trial = one config
    params = {
        "gbm__n_estimators":  trial.suggest_int("n_estimators", 100, 400),
        "gbm__max_depth":     trial.suggest_int("max_depth", 2, 6),
        "gbm__learning_rate": trial.suggest_float("learning_rate", 0.01, 0.2, log=True),
    }
    pipe.set_params(**params)                   # tune the WHOLE pipeline
    return cross_val_score(pipe, X, y, cv=5, scoring="roc_auc").mean()

study = optuna.create_study(direction="maximize")
study.optimize(objective, n_trials=50)        # each trial -> an MLflow run
print(study.best_params, study.best_value)
