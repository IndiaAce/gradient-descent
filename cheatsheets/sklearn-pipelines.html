§ 1 · WHY PIPELINES
4 RECIPES · THE CASE

Ship one object, not a pile of cells.

1.1law

A Pipeline makes leakage structurally impossible.

When cross-validation re-fits the pipeline on each fold, every preprocessing step learns from that fold's training rows only. You stop relying on remembering to split first — the architecture enforces it.

1.2core

One .joblib carries preprocessing + model together.

No separate scaler to version, lose, or mismatch at inference. The gap between "works in the notebook" and "works at the endpoint" closes.

from sklearn.pipeline import Pipeline pipe = Pipeline([("pre", pre), ("gbm", model)]) pipe.fit(X_tr, y_tr); joblib.dump(pipe, "pipe.joblib")
1.3tip

Name your steps — you'll address them later.

The string names ("pre", "gbm") become the prefix for tuning nested params and for swapping steps. Pick them deliberately.

1.4tip

make_pipeline for throwaway, Pipeline for keeps.

make_pipeline auto-names steps (lowercased class name) — fine for a quick test, painful when you need to reference them. Use explicit Pipeline for anything you'll tune.

§ 2 · COLUMNTRANSFORMER
5 RECIPES · ROUTING COLUMNS

Different columns, different treatment.

2.1core

The canonical numeric + categorical split.

Scale numerics, one-hot nominals, concatenate. This is 90% of tabular preprocessing.

from sklearn.compose import ColumnTransformer pre = ColumnTransformer([ ("num", StandardScaler(), numeric), ("cat", OneHotEncoder(handle_unknown="ignore"), nominal), ])
2.2law

Always set handle_unknown="ignore" on OneHotEncoder.

An unseen .zip TLD at inference would otherwise raise. With ignore it encodes to all-zeros — degraded, but alive.

2.3tip

Select columns by dtype, not by hand.

make_column_selector picks columns by type or name regex — the transformer keeps working as your feature table grows.

from sklearn.compose import make_column_selector as sel ("num", StandardScaler(), sel(dtype_include="number"))
2.4tip

Use remainder="passthrough" for already-clean columns.

Booleans and pre-encoded ordinals don't need a transformer. remainder="passthrough" keeps them; the default "drop" silently discards them — a classic surprise.

2.5kit

A text branch is just another transformer.

Slot a vectorizer onto a single text column alongside numeric/categorical. This is how the description-embedding feature joins Calendra.

("txt", TfidfVectorizer(max_features=300), "subject")

§ 3 · THE step__param GRAMMAR
3 RECIPES · REACHING INSIDE

Address anything by name.

3.1core

Double underscore = "reach into this step".

gbm__max_depth sets max_depth on the step named gbm. This is the grammar GridSearchCV, RandomizedSearchCV, and Optuna all use.

pipe.set_params(gbm__n_estimators=300, gbm__max_depth=4)
3.2tip

Chain it deeper for nested transformers.

Inside a ColumnTransformer named pre, target a sub-step: pre__num__with_mean=False. The path mirrors the structure.

3.3tip

Swap a whole step by assigning to its name.

You can replace an estimator entirely in a search — e.g. test GBM vs. RandomForest — by putting the step name itself in the param grid.

param_grid = {"gbm": [GradientBoostingClassifier(), RandomForestClassifier()]}

§ 4 · PERSISTENCE
4 RECIPES · JOBLIB + SERVING

Dump it, reload it, predict.

4.1core

joblib over pickle for anything with numpy.

joblib.dump handles large numpy arrays far more efficiently than the stdlib pickler. The reloaded pipeline scales raw input for you.

joblib.dump(pipe, "models/calendra_pipeline.joblib") pipe = joblib.load("models/calendra_pipeline.joblib") pipe.predict(raw_df) # no manual scaling
4.2law

Pin your sklearn version in the lockfile.

A pipeline pickled under 1.5 may warn or break under 1.8. The pyproject.toml + lock is the contract between train-time and serve-time.

4.3tip

Log the pipeline to MLflow, not just the model.

mlflow.sklearn.log_model(pipe, "model") stores the whole estimator. The registry then serves preprocessing + model as a unit.

4.4kit

Inference parity is free once it's one object.

The FastAPI /predict endpoint loads the joblib and calls .predict on the raw feature dict. No scaling code to keep in sync with training.

§ 5 · TUNING IN-PLACE
4 RECIPES · CV, GRID, OPTUNA

Search the whole pipeline.

5.1law

Cross-validate the pipeline, never the model alone.

Passing pipe to cross_val_score runs preprocessing inside each fold — the only honest number.

cv = cross_val_score(pipe, X, y, cv=5, scoring="roc_auc")
5.2core

GridSearchCV with step-prefixed params.

The grid keys are step__param paths; the best estimator is a refit pipeline ready to dump.

grid = {"gbm__max_depth": [2, 3, 4]} gs = GridSearchCV(pipe, grid, scoring="roc_auc", cv=5)
5.3kit

Optuna for ranges too big to grid.

The objective sets params on the pipeline and returns the CV mean. Each trial is leak-free because the pipeline is.

def objective(trial): pipe.set_params(gbm__max_depth=trial.suggest_int("d", 2, 6)) return cross_val_score(pipe, X, y, cv=5, scoring="roc_auc").mean()
5.4tip

Wire each trial to an MLflow child run.

Wrap the study in a parent run and each trial in start_run(nested=True). The search tree becomes sortable history. See topics/mlflow.

§ 6 · ANTI-PATTERNS
4 RECIPES · WHAT INFLATES SCORES

The traps that lie to you.

6.1trap

Fitting the scaler before the split.

The original sin. scaler.fit(X) then train_test_split leaks test statistics into training. The pipeline exists to make this impossible — don't reach around it.

6.2trap

SMOTE / resampling outside the CV fold.

Oversampling before cross-validation leaks synthetic neighbors across folds. Use imblearn.pipeline.Pipeline so resampling happens per-fold, train-side only.

6.3trap

Target-derived features computed on the full set.

Mean-encoding a category by its target rate, computed before the split, is leakage even inside a pipeline if done in a raw cell. Wrap it as a transformer that fits per fold.

6.4trap

Trusting a single train/test number.

One split is one sample of the model's variance. Report the 5-fold mean and std. A 0.978 with ±0.03 std is a different story than ±0.004.
