Jupyter for ML.

Forty-two opinions, sharply held. The things that separate a notebook you'd actually share from one you'd quietly close before anyone sees it. Setup, cell discipline, autoreload, reproducibility, MLflow integration, plotting, ML-specific tooling, and the anti-patterns that quietly destroy your experiments. Every section has runnable code; copy what you need.

§ 1 · SETUP
5 RECIPES · ENV, KERNELS, INSTALL

Get the foundations right.

1.1core

One venv per project. Install Jupyter inside it.

Global Jupyter installs leak dependencies between projects. Every project gets its own virtual environment and installs Jupyter as a project dependency, never globally. Use uv if you're starting fresh — it's an order of magnitude faster than pip and handles locking cleanly.

# inside the project root uv venv source .venv/bin/activate uv pip install jupyter ipykernel
1.2core

Register the kernel so it shows up by project name.

Without this, the kernel dropdown shows generic Python 3 entries and you'll launch the wrong env. Register every project's kernel under its own name; the UI list becomes self-describing.

python -m ipykernel install --user --name calendra --display-name "Python (calendra)"
1.3lock

Lock with pyproject.toml, never pip freeze.

pip freeze dumps the full transitive tree and rots fast. A pyproject.toml + lockfile (uv.lock or poetry.lock) keeps direct deps human-readable and resolves the rest deterministically. Future-you will thank present-you.

1.4kit

The baseline stack for any ML notebook.

This is the floor — install these eagerly and you won't break flow halfway through a session looking for a missing import.

uv pip install jupyter ipykernel \ numpy pandas scikit-learn matplotlib seaborn \ mlflow joblib tqdm rich \ ipywidgets jupyterlab-widgets
1.5config

JupyterLab over classic Notebook. Always.

JupyterLab gives you a real file tree, split views, drag-and-drop cell reordering, and proper markdown rendering. The classic interface is a dead end. jupyter lab, not jupyter notebook.

§ 2 · CELL DISCIPLINE
7 RECIPES · ORDER, SCOPE, STATE

How to keep notebooks honest.

2.1law

Restart-and-run-all before you commit.

The single most important habit. If your notebook depends on a cell you ran 30 minutes ago and have since edited, your "results" are fiction. Restart kernel → Run All → confirm everything still works. Only then commit or share.

Keyboard shortcut: Esc then 0 0 (zero twice) to restart. Esc then Shift+L to enable line numbers.

2.2structure

The top cell is always: imports + config + seeds.

One cell that sets up the world. Imports, plotting config, random seeds, environment variables. Nothing else lives there. When you reopen the notebook tomorrow, Run All starts from a known state.

# cell 1 — setup. always. import os, numpy as np, pandas as pd import matplotlib.pyplot as plt from pathlib import Path np.random.seed(42) plt.rcParams["figure.figsize"] = (10, 6) plt.rcParams["figure.dpi"] = 100 pd.options.display.max_columns = 100 DATA = Path("./data")
2.3scope

One idea per cell. Don't golf.

Cells are paragraphs, not essays. If a cell exceeds ~25 lines, you've probably mixed concepts. Split it; you'll thank yourself when re-running a single step.

2.4hygiene

Use markdown headers liberally as section anchors.

JupyterLab renders # H1 through ### H3 as a navigable table of contents (left rail). Treat headers like section dividers in a paper — your future self scrolls less.

2.5state

Never use del to fix hidden state. Restart instead.

If your notebook is misbehaving because some variable lingers from an earlier run, del var_name is a band-aid that hides the real problem. Restart the kernel. If that breaks the notebook, the notebook is broken — better to find out now.

2.6flow

Don't print(df) on a 10k-row DataFrame. Sample.

Rendering 10,000 rows blocks the kernel and clutters the output. Use df.head(), df.sample(20), or df.tail(). For inspecting columns, df.info() and df.describe(include="all") beat raw prints.

2.7order

Trust the execution counter, not the cell order.

The little [7]: in the gutter is the truth. If you see [3] below [12], cells were run out of order. Either restart-and-run-all or accept that your notebook is now a creative non-fiction piece.

§ 3 · AUTORELOAD
3 RECIPES · LOCAL MODULES

Notebooks are scratch. Modules are the artifact.

3.1magic

Enable autoreload in cell 2.

The moment you import a local Python module, you'll want autoreload. Edit the .py file, the notebook picks up the changes on next execution. No more kernel restarts every time you tweak a helper.

%load_ext autoreload %autoreload 2
3.2arch

Keep helper code in .py files, not in cells.

Cells are for exploration and presentation. Functions, classes, and pipelines belong in importable Python modules. Your notebook becomes a thin orchestration layer — far easier to review, version, and reuse.

# notebook from calendra.features import build_feature_matrix from calendra.metrics import evaluate X, y = build_feature_matrix(events) metrics = evaluate(model, X, y)
3.3caveat

Autoreload won't reload class state. Restart when changing dataclasses.

If you redefine a dataclass field or change a class hierarchy, existing instances keep their old type. Either restart the kernel or re-instantiate. Knowing the limitation up-front saves an hour of debugging.

§ 4 · REPRODUCIBILITY
5 RECIPES · SEEDS, PATHS, OUTPUTS

The notebook should produce the same numbers next month.

4.1seed

Seed everything. Once. At the top.

Every library that touches randomness has its own seed source. Set them all in the setup cell. Pass random_state=42 to every sklearn function that takes it.

import random, numpy as np random.seed(42) np.random.seed(42) # if you use torch: # import torch; torch.manual_seed(42) # sklearn: always pass random_state=42
4.2paths

Use pathlib. Never hardcode strings.

Hardcoded "/Users/you/projects/calendra/data/..." breaks the moment anyone else opens the notebook. Anchor paths to a Path rooted at the project, and treat the rest as joins.

from pathlib import Path ROOT = Path(__file__).resolve().parent if "__file__" in dir() else Path.cwd() DATA = ROOT / "data" MODEL = ROOT / "models" df = pd.read_parquet(DATA / "train.parquet")
4.3outputs

Save training-notebook outputs. Strip exploration-notebook outputs.

Two different rules. The training notebook is a record — its outputs (metrics, plots, run IDs) are the deliverable. Commit it with outputs intact. Exploration notebooks are scratch — strip outputs before committing with nbstripout, so the diff is just code.

uv pip install nbstripout nbstripout --install # now outputs are auto-stripped on git add for excluded patterns
4.4data

Split before you do any feature engineering.

The most common source of silent leakage. If you fit an OneHotEncoder or compute a mean for imputation on the full dataset before splitting, your test set has leaked. Always: load → split → engineer on train → transform on test.

4.5env

Record the actual library versions at the top.

Six months from now, "it worked on my machine" is not a debugging tool. Print versions into the first output cell.

import sklearn, numpy as np, pandas as pd import mlflow print(f"sklearn={sklearn.__version__}, np={np.__version__}, " f"pd={pd.__version__}, mlflow={mlflow.__version__}")

§ 5 · MLFLOW IN NOTEBOOKS
4 RECIPES · TRACKING + ARTIFACTS

Wrap every training cell.

5.1start

Three lines turn an ad-hoc training cell into a tracked run.

The minimum viable wrap. Drop into your existing training cell — no other changes required.

import mlflow mlflow.set_experiment("calendra-gbm") with mlflow.start_run(run_name="baseline"): model.fit(X_train, y_train) mlflow.log_params(params) mlflow.log_metric("auc", auc_score) mlflow.sklearn.log_model(model, "model")
5.2auto

Use autolog for one-line tracking on common frameworks.

If you're using sklearn / xgboost / lightgbm / pytorch, MLflow can log params and metrics automatically. Useful when you want tracking but don't want to write logging code yet.

mlflow.sklearn.autolog() # enables auto-logging for all subsequent runs # now any model.fit(...) call inside a run is logged automatically
5.3compare

Name your runs so the UI is actually useful.

run_name="run_42_4a8b" gives you a sortable table of meaningless strings. run_name="shallow-fast-no-embed" gives you a sortable table you can actually read.

5.4ui

Open the UI from the notebook with a magic.

Save yourself the terminal trip. The MLflow UI runs in a backgrounded process and renders in an iframe right inside Jupyter.

%load_ext mlflow_extras # pip install mlflow-extras %mlflow ui # or just shell-out from the notebook !mlflow ui --port 5000 &

§ 6 · PERFORMANCE
6 RECIPES · TIME + MEMORY + DTYPES

When the notebook gets slow.

6.1time

%time for one-shot. %timeit for repeated.

Two magics. %time on a single cell or line; %timeit runs it many times and reports mean / stdev. Use %%time / %%timeit for whole cells.

%timeit df.groupby("sender").agg({"url_count": "sum"}) # 12.3 ms ± 0.4 ms per loop (mean ± std. dev. of 7 runs, 100 loops each)
6.2profile

%prun when %timeit says "too slow" and you need to know why.

Function-level profiler. Sort by cumulative time, look at the top 5 — that's where the wall-clock lives.

%prun build_feature_matrix(events) # shows which functions burned the time
6.3mem

Check memory before you OOM.

df.memory_usage(deep=True).sum() / 1e9 gives you GB. If you're loading a 4GB table on a 16GB laptop, you have 25% of RAM in one variable. Cast dtypes before going further.

df.memory_usage(deep=True).sum() / 1e9 # GB df.info(memory_usage="deep")
6.4dtypes

Specify dtype= on read. Don't load as object.

By default pandas reads strings as object dtype — slow, memory-hungry. Specify dtype or use "category" for low-cardinality columns. Same data, often 5–10× less memory.

df = pd.read_csv("events.csv", dtype={ "sender_domain": "category", "org_tier": "int8", "is_malicious": "bool", })
6.5big

For > 1 GB tables: reach for polars or pyarrow.

Pandas is great for the gigabyte you're holding in your head. Beyond that, Polars (zero-copy, lazy, multithreaded) or PyArrow streaming kills the bottleneck. The API gap closes faster every release.

6.6cache

Cache expensive intermediate computations to disk.

Joblib's memoize keeps deterministic, expensive function results between sessions. Saves 5 minutes of re-running every time you reopen the notebook.

from joblib import Memory mem = Memory(location=".cache", verbose=0) @mem.cache def expensive_embed(texts): return model.encode(texts)

§ 7 · ML-SPECIFIC
7 RECIPES · PIPELINES, SPLITS, SERIALIZATION

The practitioner recipes.

7.1law

Split before any preprocessing. Always.

The cardinal sin. If you fit a scaler or impute a mean on the full dataset before splitting, the test set has leaked into the model. The training–test split is the first thing you do, not the last.

7.2pipeline

Every project ships as a Pipeline.

Even for a single model. Pipelines wrap preprocessing + model into one object that can .fit, .predict, and .save together. No more "what scaler did I use in training?" mysteries.

from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler pipe = Pipeline([ ("scale", StandardScaler()), ("model", GradientBoostingClassifier(random_state=42)), ]) pipe.fit(X_train, y_train)
7.3columns

For mixed column types: ColumnTransformer.

Numeric columns need scaling. Categorical columns need encoding. ColumnTransformer applies different transforms to different column subsets and stitches them back together.

from sklearn.compose import ColumnTransformer from sklearn.preprocessing import StandardScaler, OneHotEncoder prep = ColumnTransformer([ ("num", StandardScaler(), ["url_count", "attendees"]), ("cat", OneHotEncoder(sparse_output=False), ["org_tier"]), ]) pipe = Pipeline([("prep", prep), ("clf", model)])
7.4save

Use joblib for sklearn artifacts, never raw pickle.

Joblib is faster, handles numpy arrays better, and is the sklearn-recommended path. For sharing or production, use MLflow's log_model which wraps joblib with metadata and a clean reload path.

import joblib joblib.dump(pipe, "models/v4.1.joblib") loaded = joblib.load("models/v4.1.joblib")
7.5imbalance

Always check class balance before training.

A 99% accurate "model" that predicts the majority class is a one-liner. Check y_train.value_counts(normalize=True) first. If imbalanced, use class_weight="balanced" or stratified sampling, and report F1 or PR-AUC instead of accuracy.

7.6cv

Cross-validate through the pipeline.

If you cross-validate only the model, preprocessing leaks across folds. Use cross_val_score on the full pipeline so each fold re-fits the preprocessor on its own training data.

from sklearn.model_selection import cross_val_score scores = cross_val_score(pipe, X, y, cv=5, scoring="f1") print(f"f1: {scores.mean():.3f} ± {scores.std():.3f}")
7.7display

For DataFrames you want to read, style them.

Heatmap-colored numerical DataFrames are dramatically easier to scan than raw tables. Especially for confusion matrices and feature importance tables.

df.style.background_gradient(cmap="YlOrRd", axis=0) \ .format("{:.3f}")

§ 8 · ANTI-PATTERNS
5 RECIPES · WHAT TO NEVER DO

The quiet destroyers.

8.1danger

Don't store API keys in cells. Use os.environ.

If you commit the notebook, the key is in your git history forever. os.environ["ANTHROPIC_API_KEY"] + a .env in .gitignore. Or use a secrets manager. Anything but a string literal in a cell.

8.2danger

Don't run cells out of order then commit the notebook.

The execution counter tells the truth. If [3] is below [12], restart-and-run-all before commit. Otherwise the file you commit doesn't actually produce its own outputs.

8.3danger

Don't mix training and evaluation in the same cell.

The second the cell is too big to reason about in one screen, you'll start re-running training accidentally when you only wanted to re-evaluate. Always: one cell trains and saves; the next loads and evaluates.

8.4danger

Don't trust .head() as your only sanity check.

The first 5 rows can hide all kinds of nightmares — sorted data, NaN clusters, leakage. Use .sample(20), .describe(), and quick histograms on every column before training.

8.5danger

Don't load the full dataset for EDA. Sample first.

For initial exploration, df = pd.read_parquet(p).sample(n=10_000). You'll iterate ten times faster, then run the final pipeline on the full set. The exception: when answering questions about distributional tails, where you need everything.
